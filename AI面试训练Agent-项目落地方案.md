# AI 面试训练 Agent · 项目落地方案

---

## 一、总判断

- **最推荐项目方向**：AI 面试训练 Agent（Agent Team 架构）
- **为什么这个方向成立**：
  1. 你本人就是目标用户——转行求职者，对"面试后得不到反馈"这个痛点有真实体感
  2. 充分利用你已有的 Coze 智能体搭建经验，是升级版而非从零开始
  3. Agent Team 架构是 AI PM 面试高频考点，做过一遍比看十篇文章管用
  4. 不依赖任何内部数据——公开 JD + 你自己的面试经历就能做评测集
  5. 面试时故事线无懈可击："我是一个转行者，我发现面试反馈是最大黑箱，所以我设计了这个产品"
- **当前最大风险**：如果你只停留在 Coze 搭个对话 bot 的阶段，没有评测闭环，面试官会认为你只是"调了个 prompt"
- **不建议主打的方向**：股票交易项目（除非按上一轮的评测体系彻底迭代——金融场景对数据权威性和风险合规要求太高，作为个人项目防御难度大）

### 项目底层逻辑

```text
从【面试者想知道回答好在哪、差在哪、怎么改】这个具体用户任务出发
→ 选择【Agent Team（JD解析 + 面试官 + 评估官 + 教练 + Judge复核）】
→ 设计【30条面试问答评测集 + 四层MECE评分标准】
→ 跑评测集，定位【评估笼统不指扣分点 / 教练建议与评估脱节】等链路问题
→ 针对低分维度迭代【评估官prompt细化 / Judge复核规则 / 教练输出格式约束】
→ 用同一评测集复测，观察【评语可执行性 / 建议与评估一致性】提升
```

## 三、推荐项目落地方案

### 1. 项目一句话定位

> 面向**转行求职者**在**面试后无法获得结构化反馈**的痛点，设计 **AI 面试训练 Agent**，通过 **JD解析→面试官提问→评估官多维打分→教练生成改进建议→Judge复核** 的 Agent Team 工作流，实现**可量化、可追溯、可迭代的面试诊断**。

### 2. 真实业务基础

- **用户是谁**：转行求职者（和你一样，换了赛道的人）——他们对目标行业的面试套路不熟，不知道自己的回答在什么水平
- **场景是什么**：准备面试期间，找人 mock 成本高（时间/人情），自己练不知道好坏
- **原始流程/材料**：
  - 真实岗位 JD（你投过的那些 AI PM / AI 产品运营岗位）
  - 你自己的面试经历（面过的、挂过的、拿了 offer 的）
  - 网上公开的面经和模拟面试视频
- **用户真实参与点**：你就是用户本人，也在大量面试——可以直接用自己的真实问答做素材
- **可以补做的证据**：
  - 收集 5-10 个 AI PM 岗位 JD 作为输入样本
  - 记录你自己的面试问答（录下来转文字，或者事后回忆记录）
  - 邀请 2-3 个同样在找工作的朋友贡献他们的面试问答
  - 手动标注每个回答的四维度分数

### 3. 用户、场景、痛点

| 维度 | 内容 |
|---|---|
| **用户** | 转行求职者（非应届、跨行业），已有工作经验但面对新行业面试套路不熟 |
| **场景** | 投递后收到面试邀请，需要高效准备；或面试后感觉不好但不知道问题出在哪 |
| **核心痛点** | ① 找不到人对练，mock 成本高（付费辅导一次 200-500 元）；② 自己练完不知道哪里扣分，只能"感觉不太好"；③ 网上搜的面经跟自己的岗位/经历不匹配 |
| **为什么需要 AI** | 传统方案是找朋友 mock 或买付费辅导——朋友给不出专业评估，付费辅导贵且不可持续。AI 可以：理解任意 JD 并生成针对性问题、按统一标准反复评估不疲劳、每次给出结构化反馈和示范 |
| **必须人工确认的环节** | ① 最终面试建议仅供参考，不可替代真实面试官的判断；② 示范回答需标注"此为AI示例，实际表达需结合本人真实经历"；③ 不提供"包过""押题"等承诺性输出 |

### 4. AI 介入环节

```text
用户输入（目标JD + 个人简历/背景）
→ Task Specification：拆解JD → 提取核心能力要求 → 生成5个针对性面试题
→ Context Selection：加载该岗位类型的面试评估标准、常见追问模式
→ 面试官 Agent：逐题提问，根据用户上一条回答动态追问（模拟真实面试压力）
→ 评估官 Agent：按四层 MECE rubric 对每条回答逐维度打分 + 逐条写具体评语
→ 教练 Agent：根据低分维度生成可执行的改进建议 + 改写示范
→ Judge Agent：复核评估官评语是否自洽（检查"分数低但评语没指出具体扣分点""教练建议和评估结论矛盾"）
→ 输出：总分数 + 各维度雷达图 + 每条回答的详细诊断 + 改进建议 + 示范回答
→ Human-in-the-loop：标注"AI评估可能存在的偏差：① 无法模拟真实面试官的情绪感知；② 示范回答需结合本人经历修改"
```

**每个节点的输入、输出、评价标准和失败表现**：

| 节点 | 输入 | 输出 | 本节点的评价标准 | 失败表现 |
|---|---|---|---|---|
| Task Specification | JD原文 + 用户背景 | 拆解后的能力要求 + 5道面试题 | 题目是否覆盖JD核心要求 | 生成的题目与JD无关，或只问通用问题 |
| 面试官 Agent | 当前题目 + 用户上条回答 + 对话历史 | 下一道题或追问 | 追问是否合理、是否有压迫感 | 追问跑题、不追问随便过 |
| 评估官 Agent | 题目 + 用户回答 + JD要求 | 四维度分数 + 逐维评语 | 评语是否具体到扣分点 | "回答不够好"这种笼统评语 |
| 教练 Agent | 低分维度 + 原始回答 + JD要求 | 改进建议 + 改写示范 | 建议是否可执行、示范是否具体 | 建议是"下次回答好一点"，示范跟用户原回答无关 |
| Judge Agent | 评估官输出 + 教练输出 | 一致性判断 + 矛盾标记 | 是否发现真正的逻辑矛盾 | 漏检矛盾、或误报正常的合理差异 |

### 5. 产品链路与技术架构

**架构选择：Agent Team**

选择理由：
- 面试官、评估官、教练三个角色的 **prompt 目标完全不同**——面试官追求追问和压力感，评估官追求评分准确性和评语具体性，教练追求建议可执行性——塞进一个 prompt 会互相干扰
- 分开后可以**独立迭代**：评估官评分不准就只改评估官 prompt，不影响面试官的追问逻辑
- 增加 Judge 复核节点后，形成**内部质量检查闭环**

为什么不用更简单的 Fixed Workflow：
- 面试官的追问是**动态的**——用户上一条回答的质量决定了是否追问、追问什么
- Fixed Workflow 适合路径稳定的任务（比如简历诊断），不适合需要动态交互的对话场景

为什么不用更复杂的架构：
- 不需要 Sub-agent——每个 Agent 的任务范围在启动时就能定义清楚，不需要运行时动态创建
- 不需要并行调用——面试是线性的，先问再评再教，没有并行需求

**完整链路**：

```text
第一步：用户粘贴目标JD + 上传简历/简述背景
第二步：Task Specification 拆解 JD → 提取核心能力（如"用户洞察""数据分析""跨团队沟通"）→ 每个能力生成1-2道面试题
第三步：面试官Agent 逐题提问，根据回答质量决定追问或进入下一题
第四步：评估官Agent 对每条回答按四维度打分 + 标注具体扣分点
第五步：教练Agent 对低分维度生成改进建议 + 改写示范
第六步：JudgeAgent 复核 → 发现矛盾则标记，返回评估官/教练修正
第七步：输出完整诊断报告
```

**Harness 映射**：

| Harness 节点 | 本项目怎么体现 | 输入 | 输出 | 失败表现 |
|---|---|---|---|---|
| Task Specification | JD拆解 → 能力提取 → 题目生成 | JD原文 | 能力列表 + 题库 | 题目与JD无关 |
| Context Selection | 根据岗位类型加载评估标准 | 岗位类型标签 | 该岗位的评分权重配置 | 用错了岗位类型的评估标准 |
| Tool / Agent Routing | 固定路由：面试官 → 评估官 → 教练 → Judge | 上一节点的输出 | 下一节点的输入 | 不需要路由决策（链路固定），但 Judge 触发修正时需要回溯 |
| State / Tracing | 记录每道题、每次回答、每轮评分 | 对话轮次 | case_id + 题目 + 回答 + 分数 + 评语 + badcase类型 | 对话长了之后丢失前面的评分上下文 |
| Verification / Evals | Judge 复核 + 人工抽检 + 评测集复测 | 评估官输出 + 教练输出 | 一致性判断 + badcase标记 | Judge 自身也可能误判 |
| Human-in-the-loop | 输出标注"AI评估仅供参考"，不替代真实面试 | — | 风险提示 | 用户过度信任AI评分 |

### 6. 评测集与自动化评估设计

#### 6.1 评估目的与流程

**评估目的**：内部迭代——验证 Agent Team 各节点的输出质量，定位最弱的节点，指导 prompt 优化方向

**评估流程**：

```text
准备 30 条面试问答 eval cases（标注标准答案）
→ 跑 baseline 链路（初始版 prompt）
→ 按 MECE rubric 打分
→ 找到低分维度
→ 查看 tracing 定位是哪个 Agent 节点的问题
→ 修改该节点的 prompt / 输出格式约束
→ 用同一评测集复测
→ 对比分数变化
```

#### 6.2 评测集来源与取舍

| 来源 | 优点 | 风险 | 本项目怎么用 |
|---|---|---|---|
| 自己的真实面试问答 | 最贴近真实场景，有体感 | 样本量小（取决于你面了多少），偏个人经历 | 作为核心素材，至少贡献 10 条 |
| 转行朋友的真实面试经历 | 增加样本多样性 | 回忆偏差，质量参差不齐 | 邀请 2-3 人贡献，预期 10-15 条 |
| 网上公开面经（B站/小红书/牛客） | 量大，覆盖不同行业和水平 | 可能被润色过，不完全是真实问答 | 补足到 30 条，优先选有视频的（能看到表达细节） |
| 自己构造低质量回答 | 能覆盖极端 case | 可能不够自然 | 故意构造 5 条"典型烂回答"（空话/跑题/自相矛盾）做边界压测 |

#### 6.3 MECE 评分标准

| 一级维度 | 二级指标 | 怎么打分 | 低分说明什么 | 对应链路节点 |
|---|---|---|---|---|
| **内容匹配度（0-3分）** | 回答是否命中JD核心能力要求 | 3=完全命中且有深度；2=命中但浅；1=擦边；0=完全跑题 | 用户不理解岗位到底要什么能力 | Task Specification / 面试官Agent题目设计 |
| **结构清晰度（0-3分）** | 是否有逻辑结构（STAR/总分总/因果链） | 3=结构清晰，听一遍就能复述；2=有结构但不够紧凑；1=散乱但信息有用；0=完全没有结构 | 表达组织能力弱，面试官抓不住重点 | 评估官Agent 结构判断 |
| **证据支撑度（0-3分）** | 是否有具体数据、事例、项目支撑，而非空话 | 3=有具体数字/案例/可追溯细节；2=有例子但笼统；1=只有结论没有例子；0=全是套话 | 经验真实性存疑，或缺乏量化思维 | 评估官Agent 证据判断 |
| **表达可信度（0-3分）** | 是否有夸大、前后矛盾、模糊表述 | 3=表达谨慎，有边界意识，不夸大；2=整体可信但个别表述偏大；1=有多处模糊或夸大；0=明显矛盾或编造 | 有过度包装嫌疑，面试官一旦追问就穿帮 | Judge Agent 矛盾检测 |

**总分为 0-12 分，各维度独立评分，不取加权平均**——因为四个维度反映的是不同链路节点的问题，加权平均会掩盖问题。

#### 6.4 自动化评估执行方式

- **第一版人工标注样本数**：30 条。每条你需要标注"这道题在考什么能力""一个满分回答应该包含什么""该回答在四个维度各自得几分及为什么"
- **LLM-as-a-Judge 是否使用**：是。但第一版 30 条全部人工标注，作为 Judge prompt 的校准基准。之后新样本用 LLM 批量打分，每 20 条抽 5 条人工复核，如果 LLM 和人工评分偏离超过 1 分，校准 Judge prompt
- **Judge prompt 校准方式**：把人工标注的 30 条作为 few-shot example 放进 Judge prompt，让 Judge 先学习评分口径
- **抽样人工复核比例**：20%（每 20 条抽 4 条）
- **tracing 表字段**：

```text
case_id / 用户背景 / JD类型 / 题目 / 用户回答 / 
Task Spec输出（是否命中JD核心） / 
面试官追问（是否合理） / 
评估官四维度分数 / 评估官逐维评语 / 
教练改进建议 / 教练示范 / 
Judge复核结果（是否发现矛盾） / 
badcase类型 / 归因节点 / 修改动作 / 复测分数
```

### 7. Badcase 与优化策略

#### Badcase A：基础问题 — 评估笼统，不指具体扣分点

- **低分维度**：内容匹配度（与真实水平无关——是评估质量的问题）
- **用户/业务现象**：评估官对一条明显跑题的回答给出"回答不够好，建议改进"的评语，没有说哪里跑题、跑了多远
- **问题表现**：用户看完评语不知道怎么改，因为评语本身没有信息量
- **tracing 看到什么**：评估官 prompt 里只有"请对回答打分并写评语"，没有要求区分"哪里好/哪里不好/具体怎么改"
- **链路归因**：评估官 Agent 的 prompt 约束不够细
- **修改哪个节点**：评估官 prompt 输出格式
- **优化动作**：在评估官 prompt 中增加强制输出格式——"每条评语必须包含：① 扣分点：具体哪句话/哪个信息缺失导致扣分；② 对比：一个得分更高的回答会怎么说；③ 示范：改写其中一句话作为示例"
- **复测指标**：评语可执行性（人工判断评语是否能让用户知道"改哪里"）
- **预期分数变化**：评语可执行性从平均 1.5 分提升至 2.8 分（1-5 分量表）

#### Badcase B：进阶问题 — 教练建议与评估脱节

- **低分维度**：证据支撑度（教练给的建议没有解决评估官发现的问题）
- **用户/业务现象**：评估官指出"回答缺乏具体数据支撑，只说了'我做了很多优化'但没有数字"，教练给了"建议在回答中加入更多细节"——也是空话，等于没给建议
- **tracing 看到什么**：评估官输出了具体扣分点，但教练 Agent 只看了用户原始回答 + 总分，没有加载评估官的具体评语。教练不知道"评估官具体说了什么"，所以只能给泛泛建议
- **链路归因**：教练 Agent 的 Context Selection 缺少评估官评语作为输入
- **修改哪个节点**：教练 Agent 的上下文加载 + Judge 的检查规则
- **优化动作**：
  ① 修改教练 Agent 的上下文：在 prompt 里强制要求先阅读评估官评语，再生成建议，建议必须直接回应评估官指出的扣分点
  ② 在 Judge 复核规则中增加一项检查："教练的每条建议是否明确对应评估官指出的至少一个扣分点？"
- **复测指标**：评估-教练一致性（Judge 判断教练建议是否回应了评估扣分点的比例）
- **预期分数变化**：评估-教练一致性从约 40% 提升至 85%+

### 8. 3-7 天可补做材料

- [ ] 在 Coze 或 Dify 上搭建一版 Agent 原型，跑通"JD 输入 → 题目生成 → 回答 → 评分 → 建议"的完整对话流程
- [ ] 用标注数据跑第一版评测，记录各维度平均分，识别 top 2 低分维度

**P1（强烈建议）**：

- [ ] 针对 top 2 低分维度，写 2 个 badcase 记录（对照上面 Badcase A/B 的结构）
- [ ] 修改对应节点 prompt，复测，记录分数变化
- [ ] 画一张完整的链路架构图（面试时可以共享屏幕讲）

**P2（加分项）**：

- [ ] 找 2-3 个朋友用你的 Agent 做一次模拟面试，收集真实用户反馈

- [ ] 准备一份面试讲解 PPT（3-5页）：项目背景 → 链路图 → 评测设计 → badcase迭代 → 关键数据

### 9. 面试追问防御

#### Q1：你做了个面试训练的Agent——市面上已经有类似产品了，你的有什么不同？

**答**：

> "大多数面试 AI 产品是'题库 + AI打分'，但打分的颗粒度很粗——基本就是一个总分加一句'回答得不错'。我做的差异在于三点：
>
> 第一，**评估维度是 MECE 的**——把面试回答拆成内容匹配、结构、证据、可信度四个独立维度，每个维度对应不同的能力缺陷。用户不是看一个总分，而是知道'我内容答对了但结构碎了'还是'我结构很好但全是空话没有证据'，这两种情况的改进方向完全不同。
>
> 第二，**评估和教练是分开的 Agent**——评估官负责诊断，教练负责开药方，中间加了一个 Judge 检查药方有没有对应诊断结果。这解决了一个常见问题：很多 AI 面试产品给的改进建议是'多准备几个例子'这种废话，因为它没有把诊断结论喂给建议生成环节。
>
> 第三，**我做了评测闭环**——我手动标注了 30 条问答，用四维 rubric 打分，然后跑我的 Agent 链路，对比 AI 评分和人工评分的偏差。针对偏差大的维度改 prompt，复测。这个闭环是大多数 demo 没有的。"

#### Q2：你怎么知道评估官的打分是准的？AI 打分会不会跟真实面试官差很远？

**答**：

> "这恰恰是我整个项目最核心的验证问题。我的做法是：
>
> 第一，**人工标注作为基准**。我自己标注了 30 条问答，每条在四个维度上打分，这是我对'好坏'的判断。然后让评估官 Agent 对同一批数据打分，看每个维度的 AI-人工偏差。第一版评估官在'表达可信度'这个维度上偏差最大——它看不出来哪些话是套话，因为它不理解中国职场语境下的'我在项目中起到了关键作用'这种表述的可信度问题。我在 prompt 里加了具体的话术示例作为判断标准，偏差下降了。
>
> 第二，**我明确说了这个评估是什么、不是什么**。我的项目定位是'帮用户发现回答中的结构化问题'——逻辑乱不乱、有没有证据、有没有前后矛盾——这些是可系统化判断的。但'面试官喜不喜欢你的气场'这不是 AI 能判断的，我也没声称能判断。边界很重要。
>
> 第三，**真实面试官也不一致**。两个面试官对同一个人同一道题的打分可能有显著差异。AI 的价值不在于替代面试官，而在于让求职者在见真人之前，先把明显的结构性问题改掉——这些问题是无论哪个面试官都会扣分的。"

#### Q3：这个项目你是一个人做的，如果有工程师和设计师跟你合作，你会怎么分工？

**答**：

> "这是考察我是否理解 PM 的角色边界。如果有一个团队：
>
> **工程师负责**：Agent 框架的工程实现（Coze/Dify 的底层对接或自建）、对话状态管理、tracing 日志系统、LLM-as-a-Judge 的批量评测管线。
>
> **设计师负责**：用户交互流程——JD 上传/粘贴的体验、评测报告的呈现方式（雷达图还是表格）、模拟面试的界面节奏感。
>
> **我（PM）负责**：
> ① 定义产品方向——做什么不做什么（比如做面试训练不做简历优化）
> ② 设计 Agent 的角色分工和协作流程——这个 Agent Team 到底要几个角色、各自输入输出是什么
> ③ 定义评测标准——四个 MECE 维度是什么、每个怎么打分、什么算好什么算差
> ④ 跑评测、看数据、定 badcase、决定迭代优先级——比如评估笼统的问题比建议脱节的问题更优先修，因为它影响面更大
> ⑤ 边界决策——什么功能必须有人工兜底、什么话不能说（不承诺面试通过率等）
>
> 我不会写 Agent 的工程代码，但我必须能看懂工程师的 tracing 日志，能根据数据决定'评估官 prompt 需要改'还是'是检索问题不是生成问题'。"

#### Q4（可能的压力追问）：你连真实面试官怎么打分的都不知道，你怎么标注标准答案？

**答**：

> "诚实说，我的标注不是'面试官会怎么打分'——我也不是面试官，确实不知道。我的标注是'作为一名求职者，我判断这条回答的质量高低'。这个标注基准有局限性，但在这个项目场景下够用——
>
> 第一，我标注的维度（结构、证据、可信度）是跨行业通用的，不依赖特定面试官偏好。
> 第二，我对我标注的每一条都写了'为什么打这个分'的理由，这个理由是透明的、可讨论的。如果有争议，争议本身就说明这个维度的评分标准需要更细化。
> 第三，如果要继续迭代，下一步是邀请有面试官经验的人（比如做招聘的 HR 朋友）来校准标注，而不是换掉整个评测框架。
>
> 这个项目定位是'帮用户补基础分'——把最明显的结构性问题改掉。至于面试官的主观偏好，这不是 AI 能也不该去模拟的。"

## 四、简历表达方向

### 项目标题

**AI 面试训练 Agent | 个人项目**

### 背景句

针对转行求职者面试后无法获得结构化反馈的痛点，设计 Agent Team 架构的 AI 面试模拟与诊断产品。

### 可替换 bullet

- **【AI 技术方案 + 产品决策】**：设计 JD 解析 → 面试官提问 → 评估官多维打分 → 教练改进建议 → Judge 一致性复核的 **Agent Team** 工作流。选择 Agent Team 而非单 prompt 的原因为：面试官追问、评估官评分、教练建议三者的 prompt 目标不同，分开设计可独立迭代且互相不干扰；增加 Judge 复核节点解决评估与建议脱节问题。

- **【评测与自动化评估】**：从个人真实面试经历和公开面经中采集 30 条面试问答，构建按**内容匹配度、结构清晰度、证据支撑度、表达可信度**四层 MECE rubric 的评测集。采用人工标注建立评分基准，再用 LLM-as-a-Judge 批量评分，每 20 条抽样 4 条人工复核校准 Judge prompt。

- **【Badcase 迭代】**：定位"评估笼统不指具体扣分点""教练建议与评估结论脱节"两类 badcase。前者通过细化评估官 prompt 的输出格式约束（强制标注扣分点 + 改写示例）解决；后者通过将评估官评语强制纳入教练的 Context Selection + Judge 增加建议-评估一致性检查规则解决。复测后评语可执行性从 X 分提升至 X 分，评估-教练一致性从 XX% 提升至 XX%。

- **【边界与兜底】**：在输出中标注 AI 评估的适用边界——可系统化判断结构、证据、可信度问题，但不模拟面试官的主观偏好和情绪感知；不提供"面试通过率预测"等无法验证的承诺性输出，示范回答标注"需结合本人真实经历调整"。

## 五、边界提醒

**可以写**：
- 设计 Agent Team 产品链路与角色分工
- 构建评测集、设计 MECE 评分标准
- 手动标注 30 条问答作为评分基准
- 使用 LLM-as-a-Judge 做批量评估
- 定位 badcase、归因到具体 Agent 节点、修改 prompt 并复测
- 在 Coze/Dify 平台搭建可交互原型

**谨慎写**（需要有对应证据）：
- "收集了 XX 位用户的反馈"——如果只是找朋友用了用，如实写"小范围用户试用"，不要造数字
- "评估准确性达到 XX%"——需要先定义"准确性"怎么算，样本是多少，基准是什么

**不能写**：
- "帮助 XX 名用户成功通过面试"——无法验证，且暗示因果关系
- "模拟真实面试官评分逻辑"——你没有面试官评分数据来源
- "接入企业内部面试题库"——你没有这个权限
- 任何声称产品上线、有用户量、有商业结果的表述

**这份方案的核心价值不在于"做了一个Agent"，而在于你通过它证明了三件事：**
1. 你能定义 AI 产品中"好"的标准（MECE rubric）
2. 你能系统性发现问题（tracing → badcase归因 → 定位节点）
3. 你能用数据驱动迭代（改prompt → 用同一评测集复测 → 分数可对比）

这三件事，就是 AI 产品经理面试中最稀缺的信号。
