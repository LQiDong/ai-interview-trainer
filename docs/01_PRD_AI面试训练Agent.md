# PRD：AI 面试训练 Agent

> **文档状态**：V1.0 | **创建日期**：2026-06-11 | **作者**：AI 产品经理面试作品集

---

## 1. 产品概述

### 1.1 产品定位

AI 面试训练 Agent 是一款面向**转行求职者**的 AI 面试模拟与诊断工具。产品通过多 Agent 协作，模拟真实面试中的"面试官追问 + 多维度评估 + 教练反馈"闭环，帮助用户获得**结构化、可行动的面试诊断报告**，而非泛泛的"你答得不错/继续努力"。

**一句话定位**：一个会追问、会打分、会给改进建议的 AI 面试教练。

### 1.2 目标用户

| 用户画像 | 典型场景 |
|---------|---------|
| 转行 AI 产品经理的求职者 | 没有 AI PM 面试经验，不知道会被问什么、怎么答 |
| 转行 AI 产品运营的求职者 | 缺乏对 AI 产品指标体系、A/B 实验设计的系统理解 |
| 转行 Agent 产品经理的求职者 | 不熟悉 Multi-Agent 架构、工具调用、记忆管理等考点 |
| 有 1-3 年经验但面试通过率低的产品经理 | 面试后得不到反馈，反复踩同样的坑 |

### 1.3 核心痛点

1. **面试后无反馈**：HR 只会说"不合适"，不告诉你哪里答得不好。
2. **自练盲区大**：对着镜子练习/录音回听，只能发现"嗯啊"口头禅，无法判断逻辑是否自洽、是否命中考点。
3. **岗位差异大**：AI PM / AI 运营 / Agent PM 的面试侧重点完全不同，通用面经不解决精准准备问题。
4. **缺乏追问压力**：真实面试中面试官会连续追问，自练无法模拟这种压力场景。

## 2. 产品边界

### 2.1 MVP 做什么

- **JD 驱动的面试模拟**：用户粘贴目标岗位 JD，系统自动解析并生成定制化面试题
- **多轮追问对话**：AI 面试官根据用户回答进行 3-5 层递进追问，模拟真实压力面
- **四维结构化评分**：评估官从四个维度独立打分并给出依据
- **教练级改进建议**：告诉用户"应该怎么答"，而非仅仅"哪里扣了分"
- **Judge 复核机制**：独立 Judge Agent 对评分和教练建议做一致性校验
- **诊断报告生成**：一份结构化的面试诊断报告（支持导出）

### 2.2 MVP 不做什么

- ❌ 不构建用户注册/登录系统（MVP 用匿名会话）
- ❌ 不构建支付/订阅体系
- ❌ 不做面试题库社区/UGC
- ❌ 不做面试课程/知识付费
- ❌ 不做真实面试官 1v1 预约
- ❌ 不做简历解析/简历优化
- ❌ 不做面试录制/回放（V1 仅保留文本对话记录）
- ❌ 不作任何"面试通过率"的承诺或预估

## 3. 核心链路

### 3.1 端到端流程

```
用户输入 JD
    ↓
[Agent 1] JD 解析器 → 结构化岗位画像
    ↓
[Agent 2] 出题官 → 生成核心题 + 追问树
    ↓
[Agent 3] 面试官 ←→ 用户（多轮追问对话）
    ↓
[Agent 4] 评估官 → 四维打分 + 逐题点评
    ↓
[Agent 5] 教练 → 改进建议 + 范例回答
    ↓
[Agent 6] Judge → 复核评分一致性 & 建议有效性
    ↓
输出：诊断报告（含雷达图数据 + 文字分析）
```

### 3.2 各环节输入/输出

| 环节 | 输入 | 输出 |
|------|------|------|
| JD 解析 | 用户粘贴的 JD 原文 | 结构化岗位画像（JSON）：岗位名称、职责、技能要求、经验要求、软技能要求 |
| 出题 | 岗位画像 | 面试题列表（含题目类型、难度、考察维度、参考答案要点） |
| 面试追问 | 面试题 + 用户回答 | 追问 + 逐轮对话记录 |
| 四维评估 | 完整对话记录 + 岗位画像 | 四维分数 + 每维度扣分/加分依据 |
| 教练建议 | 评估结果 + 对话记录 | 逐题改进建议 + 范例回答 + 通用面试策略 |
| Judge 复核 | 评估结果 + 教练建议 + 对话记录 | 复核报告：评分是否自洽、建议是否可执行、是否有遗漏 |
| 诊断报告 | 以上所有 | 结构化报告（Markdown + 雷达图数据 JSON） |

## 4. Agent Team 设计

### 4.1 架构概览

产品核心是一个 **6 Agent 协作流水线**（非端到端单模型调用），每个 Agent 有独立的人格设定、评估标准和输出格式。

```
┌─────────────────────────────────────────────┐
│              Orchestrator                   │
│         (流程编排 & 上下文管理)              │
└─────────────────────────────────────────────┘
        │         │         │         │
   ┌────▼──┐ ┌───▼───┐ ┌──▼──┐ ┌───▼───┐
   │JD解析 │ │出题官 │ │面试官│ │评估官 │
   │Agent  │ │Agent  │ │Agent│ │Agent  │
   └───────┘ └───────┘ └─────┘ └───────┘
        │         │         │         │
   ┌────▼──┐ ┌───▼───┐
   │教练   │ │Judge  │
   │Agent  │ │Agent  │
   └───────┘ └───────┘
```

### 4.2 各 Agent 详细定义

#### Agent 1：JD 解析器（JD Parser）

| 属性 | 描述 |
|------|------|
| **职责** | 将用户输入的 JD 原文转化为结构化岗位画像 |
| **输入** | 非结构化 JD 文本（可能包含公司介绍、福利描述等噪声） |
| **输出 Schema** | `{ position, industry, responsibilities[], hard_skills[], soft_skills[], experience_level, keywords[] }` |
| **核心 Prompt 策略** | 提取面试相关信号，忽略福利/公司介绍等噪声；即使 JD 写得模糊，也能推断隐含要求 |
| **容错设计** | 若 JD 信息不足，标记 `confidence: low` 并在缺失字段给出合理推断（带标注） |

#### Agent 2：出题官（Question Designer）

| 属性 | 描述 |
|------|------|
| **职责** | 基于岗位画像生成定制化面试题及追问树 |
| **输入** | 岗位画像 JSON |
| **输出 Schema** | `{ questions: [{ id, type, difficulty, dimension, question_text, follow_up_tree: [{ trigger_condition, follow_up }], reference_answer_points[] }] }` |
| **题目类型** | 行为面试题（STAR）、案例分析题、专业知识题、情境模拟题 |
| **题量控制** | MVP 每场 5-8 题，覆盖 4 个评估维度 |
| **追问树设计** | 每题预设 3-5 个追问分支，触发条件基于用户回答中的关键词/逻辑缺陷 |

#### Agent 3：面试官（Interviewer）

| 属性 | 描述 |
|------|------|
| **职责** | 与用户进行多轮对话，根据回答动态追问 |
| **输入** | 题库 + 追问树 + 用户实时回答 |
| **输出** | 完整对话记录（含追问节点标记） |
| **核心策略** | <ul><li>首轮抛出主问题</li><li>根据用户回答的深度/逻辑决定追问还是换题</li><li>用户回答含关键错误时标记为"重点追问"</li><li>单题追问不超过 4 层，总题数不超过 8 题</li><li>追问语气保持专业但有一定压力（但不会刻意打压）</li></ul> |
| **终止条件** | 题目耗尽 OR 用户主动结束 OR 对话轮次达到上限（25轮） |

#### Agent 4：评估官（Evaluator）—— 四维评分

| 维度 | 权重 | 评估内容 |
|------|------|---------|
| **逻辑结构** | 30% | 回答是否有清晰的逻辑框架（如 STAR、金字塔原理）；论点-论据是否自洽 |
| **专业深度** | 30% | 对 AI/产品概念的准确性；是否展示出对行业的理解深度；是否能区分相似概念 |
| **表达沟通** | 20% | 是否简洁有力；是否善用数据/案例支撑观点；是否有结构化的表达习惯 |
| **应变能力** | 20% | 追问下是否自乱阵脚；能否在信息不足时做出合理假设；是否敢于承认不知道并给出思考路径 |

**输出 Schema**：
```json
{
  "overall_score": 72,
  "dimension_scores": {
    "logic_structure": { "score": 75, "evidence": ["..."] },
    "professional_depth": { "score": 68, "evidence": ["..."] },
    "expression": { "score": 80, "evidence": ["..."] },
    "adaptability": { "score": 65, "evidence": ["..."] }
  },
  "per_question_analysis": [
    { "question_id": 1, "score": 70, "strengths": [], "weaknesses": [], "key_missed_points": [] }
  ]
}
```

#### Agent 5：教练（Coach）

| 属性 | 描述 |
|------|------|
| **职责** | 将评估结果转化为可行动的改进建议 |
| **输入** | 评估结果 + 对话记录 + 岗位画像 |
| **输出** | 逐题改进建议 + 范例回答 + 通用策略建议 |
| **设计原则** | <ul><li>每条改进建议必须是"可行动的"——用户读完知道明天该干什么</li><li>范例回答不是"标准答案"，而是展示更好的答法思路</li><li>区分"知识性错误"（需补充学习）和"表达性问题"（需练习改进）</li><li>给出 3 条高优先级改进项（Top 3 Actions）</li></ul> |

#### Agent 6：Judge（复核官）

| 属性 | 描述 |
|------|------|
| **职责** | 对评估官和教练的输出做独立质量复核 |
| **输入** | 评估结果 + 教练建议 + 原始对话记录 |
| **输出** | 复核报告 |
| **检查项** | <ul><li>评分是否与 evidence 一致（有无评语给分矛盾）</li><li>四维分数之间是否自洽（如逻辑满分但表达很低，是否合理）</li><li>教练建议是否针对评分中的短板（避免避重就轻）</li><li>教练建议是否可执行（避免"多练习"这类空洞建议）</li><li>是否遗漏了对话中的重要问题点</li></ul> |
| **决策权限** | Judge 发现严重分歧时，**不直接修改评分**，而是在诊断报告中标注"Judge 异议"，供用户参考 |

### 4.3 Orchestrator（编排器）

Orchestrator 不参与内容生成，仅负责：
- 按管线顺序串联 6 个 Agent
- 管理上下文窗口（对话记录可能很长，需要做压缩/摘要后传入下游 Agent）
- 异常处理（Agent 输出格式不符时的重试/降级策略）

## 5. 评测体系 & Badcase 迭代

### 5.1 为什么需要评测集

作为 AI 产品经理面试作品集，**评测体系是本项目的核心竞争力展示**。一个没有评测的 Agent 产品 = 拍脑袋调 prompt = 无法证明你真的懂 AI 产品。

### 5.2 评测集设计

#### 5.2.1 评测集构成

| 评测集 | 用途 | 规模（MVP） | 说明 |
|--------|------|------------|------|
| **JD 解析评测集** | 评估 JD 解析准确率 | 20 条 JD | 覆盖 AI PM / 运营 / Agent PM 三类岗位，手工标注 Ground Truth |
| **出题质量评测集** | 评估题目与 JD 的相关性、难度合理性 | 10 组（JD→题目） | 由 2 位 AI PM 从业者交叉标注 |
| **评估一致性评测集** | 评估四维评分与人工评分的一致性 | 15 组（对话→评分） | 同段对话由 3 人独立打分取均值作为 Ground Truth |
| **教练建议可用性评测集** | 评估建议的可行动性 | 15 组 | 由真实求职者对建议做"有用/无用"二分类标注 |

#### 5.2.2 评测指标

| 指标 | 定义 | 目标（MVP） |
|------|------|------------|
| JD 解析 F1 | 关键字段抽取的 F1 Score | ≥ 0.80 |
| 题目相关性 | 题目是否命中 JD 核心要求（人工评分 1-5） | 均值 ≥ 3.8 |
| 评估与人工评分 MAE | 总分与人工评分的平均绝对误差 | ≤ 8 分 |
| 建议可用率 | 教练建议被标注为"有用"的比例 | ≥ 70% |
| Judge 检出率 | Judge 是否能检出人工注入的评分错误 | ≥ 80% |

### 5.3 Badcase 迭代机制

```
评测发现 Badcase
    ↓
Badcase 分类入库（按 Agent + 错误类型打标签）
    ↓
分析根因（Prompt 缺陷 / 上下文不足 / 模型能力边界 / 输入质量差）
    ↓
修复方案设计（改 Prompt / 加 Few-shot / 调整 Agent 分工 / 引入外部知识）
    ↓
回归评测（确认修复不引入新 Badcase）
    ↓
更新评测集（将本次 Badcase 及变体加入评测集）
```

#### Badcase 分类体系

| 错误类型 | 示例 | 优先级 |
|---------|------|--------|
| **幻觉** | 评估官引用用户未说过的内容作为扣分依据 | P0 |
| **评分矛盾** | 各维度分数与 evidence 自相矛盾 | P0 |
| **追问跑偏** | 面试官追问与原始问题无关 | P1 |
| **建议空洞** | 教练给出"多练习就好"类无效建议 | P1 |
| **JD 误读** | 将"AI 产品运营"误解析为"AI 产品经理" | P1 |
| **难度失配** | 对初级岗位出专家级题目 | P2 |
| **语气不当** | 面试官语气过于生硬或过于随意 | P2 |

```
│    可行动、有优先级、有时间建议    │

├──────────────────────────────────┤

├──────────────────────────────────┤
│  声明：本报告由 AI 生成，不构成    │
│  面试通过保证，不替代真实面试官    │

└──────────────────────────────────┘
```

### 6.2 免责声明（每份报告强制附上）

> **重要声明**
>
> 1. 本报告由 AI 系统自动生成，仅供面试准备参考，**不构成任何形式的面试通过率承诺或保证**。
> 2. AI 评估不能替代真实面试官的专业判断——不同公司、不同面试官的偏好和评判标准差异极大。
> 3. 评分仅反映 AI 系统在当前设定下的评价，请结合自身情况理性看待。
> 4. 本系统不存储您的对话记录（MVP 阶段为本地会话，关闭即清除）。

## 7. 技术实现概要

### 7.1 技术选型建议

| 模块 | 建议方案 | 选型理由 |
|------|---------|---------|
| LLM 基座 | Claude 系列 (Opus/Sonnet) 或 GPT-4o | 多 Agent 场景对指令遵循能力要求高；Claude 在长上下文评估任务上表现好 |
| Agent 框架 | LangGraph / 自研编排层 | 需要精确控制 Agent 间数据流和异常处理，纯 LangChain 的 AgentExecutor 不可控 |
| 前端 | Next.js + Tailwind | 对话界面为主，轻量即可；支持 Markdown 报告渲染 |
| 评测管道 | Python 脚本 + 人工标注平台（Google Sheets/Airtable） | MVP 阶段不需要自建标注平台 |

### 7.2 MVP 技术约束

- 单次面试限定在 25 轮对话以内，防止 token 膨胀
- 对话记录在离开页面前做上下文压缩后再传给评估链
- 不存储用户数据（MVP 阶段所有数据存于浏览器 localStorage，关闭即清除）

## 8. MVP 交付范围 & 分期规划

### 8.1 MVP（V1.0）—— 核心闭环

**交付物**：
- [ ] 完整的 6 Agent 流水线（单次调用，非流式）
- [ ] 基础对话 UI（文本输入输出）
- [ ] 诊断报告页（Markdown 渲染 + 雷达图）
- [ ] JD 解析评测集（20 条）+ 评分
- [ ] 评估一致性评测集（15 组）+ 评分
- [ ] 诊断报告导出（复制 Markdown / 截图）

**不交付**：
- 用户系统、历史记录
- 语音输入/输出
- 流式对话（先做非流式验证效果）
- 移动端适配

### 8.2 V1.1（规划中，非 MVP）

- 流式对话 + 打字机效果（提升体验感）
- 面试场景选择（压力面/温和面/模拟群面）
- 用户手动标注"这道题答得好/不好"用于在线学习
- 历史对话记录（本地存储）

### 8.3 V2.0（远期愿景，非 MVP）

- 语音输入 + TTS 输出（模拟真实面试的语音压力）
- 表情/语气分析（回答时是否过于紧张）
- 知识库 RAG（针对特定公司/岗位的面试题库）
- 面试进步追踪（多次面试的成长曲线）

## 9. 成功指标（产品视角）

作为作品集项目，成功指标分为两层：

### 9.1 作品集展示指标

| 指标 | 说明 |
|------|------|
| 评测集覆盖率 | 核心链路每个环节均有评测集 + 可量化的评测结果 |
| Badcase 发现 & 修复数 | 展示迭代能力（目标：V1 至少记录 20+ Badcase 并完成 15+ 修复） |
| Agent 协作管线完整度 | 6 Agent 全部上线且通过 Judge 复核 |
| 产品文档完整度 | PRD + 评测方案 + 技术方案 + 复盘文档 |

## 10. 风险 & 应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| LLM 幻觉导致评估失真 | 高 | 中 | Judge 复核 + 证据溯源要求（每条评分必须有对话原文引用） |
| 用户对低评分产生负面情绪 | 中 | 低 | 教练建议先行（先给改进方案，再展示评分）；UI 上强调"分数不定义你" |
| 追问过于刁钻引发用户挫败感 | 中 | 中 | 面试官 Agent 加入"压力调节"参数，默认为中等压力；追问不超过 4 层 |
| JD 信息不全导致出题偏差 | 高 | 中 | JD 解析 Agent 主动标记低置信度字段；出题官在信息不足时主动提示用户补充 |
| 用户误将 AI 评估等同于真实面试结果 | 中 | 高 | 报告多位置嵌入免责声明；UI 层面不做分数排名/PK 等游戏化设计 |

## 附录 A：竞品分析简表

| 产品 | 优势 | 劣势 | 与本品的差异 |

## 附录 B：术语表

| 术语 | 定义 |
|------|------|
| JD | Job Description，岗位描述 |
| Agent Team | 多个具有独立角色、目标和输出规范的 AI Agent 协作完成任务 |
| 追问树 | 预设的分支追问逻辑，根据用户回答触发不同追问路径 |
| 四维评估 | 逻辑结构、专业深度、表达沟通、应变能力四个评估维度 |
| Judge | 独立复核 Agent，对评估结果做质量校验 |
| Badcase | 系统输出不符合预期的案例，用于驱动产品迭代 |
| STAR | Situation-Task-Action-Result，行为面试的标准回答框架 |
| Orchestrator | 流程编排器，负责串联 Agent 链路和管理上下文 |
