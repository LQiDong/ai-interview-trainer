# Badcase A：评估官"笼统扣分"——扣了分但说不出为什么

> **归属 Agent**：评估官 (Evaluator Agent)
> **严重度**：P1（影响评分质量和教练建议有效性）
> **发现日期**：2026-06-11
> **发现方式**：V1 评测集 AI vs 人工偏差分析
> **状态**：已修复 · 待复测

---

## 3. 问题表现

### 3.1 评分偏差：系统性偏低

| 维度 | AI | Golden | 偏差 | 问题 |
|------|-----|--------|------|------|
| 内容匹配度 | 1 | **2** | **-1** | AI 认为"不够深入"→1分，但人工认为覆盖了主要考点→2分 |
| 结构清晰度 | 1 | **2** | **-1** | AI 认为"不够清晰"→1分，但人工认为有基本时间线→2分 |
| 证据支撑度 | 0 | **1** | -1 | AI 和人工一致认为证据不足，但 AI 更低一档 |
| 表达可信度 | 2 | **2** | 0 | 一致 |

> 加权总分偏差：AI = 0.90 vs Golden = 1.75，偏差 **-0.85**

### 3.2 核心症状：扣分点不可行动

AI 评估官输出的 `deduction_points` 存在三类典型问题：

| 问题类型 | 具体表现 | 为什么是问题 |
|---------|---------|------------|
| **空洞描述** | "回答不够深入" | 教练 Agent 无法据此生成具体建议——"不够深入"到底缺什么？ |
| **循环论证** | "结构不够清晰" → 这只是在复述分数，不是扣分依据 | 扣分点应该是"缺少总-分框架"而非"结构不好" |
| **粒度太粗** | "缺乏量化数据" | 应具体到"提到'大幅降低了质检成本'但未给出基线、降低幅度、衡量方式" |

### 3.3 连锁影响

```
评估官扣分笼统
    → 教练 Agent 拿不到具体病灶
        → 教练只能生成"加强深度""优化结构"等同样笼统的建议
            → 用户读完报告仍然不知道明天该做什么
                → 产品价值归零
```

## 4. 归因节点

### 根因定位

问题出在评估官 Agent 的 **Prompt 约束层**，而非模型能力层。具体定位：

```
prompts/evaluator_agent.md
    │
    ├── 第 4 节「四维评分体系」：定义了 0-3 分的锚定标准 ✅ 已有
    │
    └── 第 5 节「约束条件」第 2 条：
        "凡是分数低于 3 分的维度，deduction_points 必须至少包含
         1 条具体扣分项"
        → 问题: "具体"的定义不够 operationalized
        → 导致 LLM 输出"回答不够深入"时认为自己满足了这个约束
```

### 根因分类

| 归因维度 | 判定 |
|---------|------|
| 是 Prompt 问题 | ✅ 是 — "具体扣分项"缺少反例和锚定示例 |
| 是模型能力问题 | ❌ 否 — V1 Prompt 本身有约束，但约束不够强 |
| 是上下文问题 | ❌ 否 — 对话记录完整传入 |
| 是输入质量问题 | ❌ 否 — 人工 Golden 可以给出明确扣分点，说明对话信息充足 |

## 5. 修改动作

### 修改对象

`prompts/evaluator_agent.md` 的「约束条件」第 2 条。

### 修改前 Prompt（原版）

```markdown
2. **扣分即指出**：凡是分数低于 3 分的维度，`deduction_points` 必须
   至少包含 1 条具体扣分项。"具体"意味着教练 Agent 仅凭该描述就能定位
   到问题并给出改进建议。不允许出现"回答不够好"这类模糊扣分。
```

### 修改后 Prompt（强化版）

```markdown
2. **扣分即指出（强制规则）**：凡是分数低于 3 分的维度，`deduction_points`
   必须至少包含 1 条具体扣分项。

   **"具体"的定义**：教练 Agent 不需要阅读原始对话，仅凭 deduction_points
   文本就能准确判断"候选人在哪句话、哪个观点上出了问题"。

   **禁止的扣分写法（反例）**：
   - ❌ "回答不够深入" → 这是复述分数，不是扣分依据
   - ❌ "结构不清晰" → 这是复述分数，不是扣分依据
   - ❌ "缺乏数据支撑" → 太笼统，应具体到"缺少什么数据"
   - ❌ "表达不够好" → 不可行动
   - ❌ "逻辑有问题" → 应指出"哪个推理环节断裂"

   **合格的扣分写法（正例）**：
   - ✅ "'提升了30%'未说明基线（优化前是多少？）和衡量口径（如何统计的？）"
   - ✅ "使用了STAR框架但缺少Result中的量化结果——只说了'上线后效果不错'"
   - ✅ "分类维度之间存在重叠——'用户侧指标'和'行为指标'不是互斥的分类"
   - ✅ "对Agent测试的回答未区分确定性测试和概率性测试——Agent输出非确定性
        使传统单元测试不适用，但候选人回答暗示两者等同"

   **自检规则**：写完每条 deduction_point 后，问自己：
   "一个不认识我的人，仅凭这句话，能否准确复现出候选人哪里答错了？"
   如果答案是否定的，这条 deduction_point 需要重写。
```

### 修改点对比

| 维度 | 原版 | 强化版 |
|------|------|--------|
| "具体"的定义 | 一句话带过 | 给出判断标准 + 自检规则 |
| 反例数量 | 1 个（"回答不够好"） | 5 个（覆盖四个维度的常见模糊写法） |
| 正例数量 | 0 个 | 4 个（覆盖四个维度的合格写法） |
| 约束强度 | "不允许出现" | "禁止" + 自检规则强制内化 |
| 可操作性 | 弱 — LLM 不理解什么叫"具体" | 强 — LLM 可以对照正反例做模式匹配 |

## 6. 复测结果

| 项目 | 状态 |
|------|------|
| Prompt 修改 | ✅ 已完成（见第 5 节） |
| 回归评测 | ⏳ **待复测** |
| 预期效果 | 评估官输出的 deduction_points 中，"笼统扣分"型表述的占比预期从 ~40% 降至 ~10% |

### 复测方案（已设计，待执行）

1. 使用 V1 评测集中 E001、E002、E004、E005（均为内容/结构维度有扣分但描述可能笼统的 case）
2. 切换到真实 LLM 模式（需 API Key），用修改后的 Prompt 重新评测
3. 人工检查 deduction_points 是否满足"不读原始对话也能定位问题"的标准
4. 计算"笼统扣分率"（模糊 deduction_points 数 / 总 deduction_points 数）

**当前未执行复测的原因**：Mock 模式下的评分是确定性 hash 计算，无法反映 Prompt 修改效果。需要真实 LLM 调用。已在 `baseline_result.csv` 中预留了 `eval_mode` 字段支持 real/mock 模式切换。

## 7. 面试讲述要点

在面试中展示这个 Badcase 时，建议强调：

1. **"我建立了一套发现问题的机制"** — 不是碰巧发现这个 Bug，而是通过评测集 + AI vs 人工对比系统性地发现了偏差模式。

2. **"我定位到了具体是哪一层出了问题"** — 不是笼统地说"评估不准"，而是定位到 Prompt 约束层的"具体扣分项定义不够 operationalized"。

3. **"修复是 precise 的，不是加更多 Prompt"** — 没有简单地加长 Prompt，而是在关键约束处增加了 5 个反例 + 4 个正例 + 1 条自检规则。修改量小但命中要害。

4. **"我知道如何验证修复是否有效"** — 复测方案已设计好，指标明确（笼统扣分率），只是等待真实 LLM 环境下执行。
