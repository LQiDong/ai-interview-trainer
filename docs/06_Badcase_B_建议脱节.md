# Badcase B：教练建议脱节——改了不该改的，漏了该改的

> **归属 Agent**：教练 Agent → 评估官 Agent（跨 Agent 对齐问题）
> **严重度**：P1（影响用户真正的改进方向判断）
> **发现日期**：2026-06-11
> **发现方式**：V1 评测集 AI vs 人工偏差分析 + Judge 检查 3（教练-评估对齐）
> **状态**：已修复 · 待复测

---

## 1. 原始输入

### 岗位画像

```
岗位：Agent产品经理（中级）
要求：负责Agent产品的运营指标体系建设，需要数据分析和A/B实验经验
```

### 面试题目

```
Q: 请描述一个你通过A/B实验验证产品假设的案例
类型: case_study
参考答案要点: 完整的A/B实验流程(假设→实验设计→实施→分析→结论);
              展示对统计显著性的理解; 体现实验伦理意识
```

### 候选人回答

```
我们有一次想优化Agent的欢迎语，就做了一个A/B测试，把原来的问候语改成
了更友好的版本，结果点击率提升了大概15%左右。我们也看了留存，好像也有
提升，但不太确定是不是因为这个改动，因为同期还上线了别的功能。
```

### 追问链（2 层）

```
Q: "15%是怎么衡量的？"
A: "我们看了一下后台的点击数据对比。"

Q: "有没有做统计显著性检验？"
A: "这个不太了解..."
```

### 人工 Golden 评分（3 人拉齐，一致性极高）

| 维度 | 分数 | 理由 |
|------|------|------|
| 内容匹配度 | **1** | 方向对但概念偏差——"看了一下后台点击数据" ≠ A/B实验 |
| 结构清晰度 | **1** | 散乱，缺少实验设计的结构化描述 |
| 证据支撑度 | **0** | **没有任何有效证据**——声称是A/B实验但不存在实验设计 |
| 表达可信度 | **1** | 对不了解的概念直接承认，但未展示思考路径 |

> 核心定性：**不是证据质量问题，是根本没有证据**。声称做的是"A/B实验"但实际只是"看了一下数据"——这是概念层面的缺失，不是表达层面的不足。

## 3. 问题表现

### 3.1 核心症状：教练建议治标不治本

| 教练说了什么 | 教练应该说什么 | 偏差本质 |
|-------------|--------------|---------|
| "使用精确的数字而非范围表述" | "**你的回答中没有A/B实验**——你只是在后台看了一次数据对比。你需要从根本上理解什么是A/B实验：随机分流、对照组、假设检验。这不是数据表达的问题，是方法论缺失的问题。" | **把概念缺失误诊为表达问题** |
| "将'大概15%'改为具体百分比" | "你无法给出精确数字，因为实验根本就没有被设计过。下次回答时诚实说'我没有做过严格的A/B实验，但我理解正确的做法是...'，这比假装有个实验更可信。" | **把虚构修补误诊为数据润色** |
| "建议深入学习A/B实验核心概念"（排在 Rank 2） | "这是你的 **Rank 1 最优先改进项**——没有A/B实验方法论，你描述的所有'数据'都不具备可信度。" | **优先级倒挂**——方法论缺失应排在首位 |

### 3.2 根因传导链

```
评估官误判：evidence=1（"数据不精确"）
    而非 evidence=0（"根本不存在有效证据"）
        ↓
教练 Agent 读取 deduction_point："数据不够精确——'大概15%'表述模糊"
        ↓
教练基于此生成建议："优化数据表达的精确度"
        ↓
用户收到建议后的行为：
    "哦，我只需要把'大概15%'改成'15.3%'就行了"
        ↓
实际效果：用户的回答从"有数据的假象"变成了"有精确数据的假象"
    → 下次面试中面试官追问"你怎么分流用户的？"
    → 用户再次暴露方法论缺失
    → 教练建议等于没给
```

### 3.3 从这个 Badcase 中发现的系统性模式

在 10 条 V1 评测集分析中，评估官在证据支撑度上的 AI vs 人工偏差呈现一个特定模式：

| Eval ID | AI 证据分 | Golden 证据分 | 偏差 | 表现 |
|---------|----------|-------------|------|------|
| E003 | 1 | **0** | **+1** | AI 把"提到了数字"当成"有证据"——低估了证据质量的下限 |
| E006 | 2 | **0** | **+2** | AI 严重高估——把"理论上的方法描述"当成了"实际验证的证据" |
| E008 | 1 | **0** | +1 | 同上——把"有分析思路"当成了"有分析证据" |

> 模式：评估官对"什么是证据"的阈值偏低。它会因为回答中出现了"数字"或"方法描述"就给证据分，而无法判断这些"数字"和"方法"是否构成了有效的证据。

## 4. 归因节点

### 根因定位

问题出在**两个 Agent 的接口处**——评估官的输出格式虽然包含了 `deduction_points` 字段，但缺少一个关键的**语义层级判断**：这个扣分属于"改进型"还是"重建型"。

| 扣分类型 | 含义 | 教练应有的响应 | 当前问题 |
|---------|------|-------------|---------|
| **改进型** | 方向对了但执行不够好（如"数据有但缺少基线"） | "补充基线、明确衡量方式" | 教练能处理 |
| **重建型** | 方向都错了/根本不存在有效内容（如"声称做了实验但实际没有"） | "你需要在面试前建立这个方法论的基础理解，而非修改措辞" | **教练无法区分** |

根因：评估官的 `deduction_points` 没有携带"扣分严重度"的语义信号。

```markdown
当前格式：
deduction_points: ["数据不够精确"]

缺少的信号：
- 这个扣分是"改进型"（微调即可）还是"重建型"（需要从零建立）？
- 教练读到"数据不够精确"会默认这是"改进型"
- 但实际上这应该触发"重建型"建议
```

### 归因分类

| 归因维度 | 判定 |
|---------|------|
| 是 Prompt 问题 | ✅ 是 — 评估官缺少"扣分严重度"的输出字段；教练缺少"区分改进型/重建型建议"的指令 |
| 是模型能力问题 | ❌ 否 — 模型有能力生成重建型建议（见 E003 的 Golden rationale 中标注者能明确写出），只是 Prompt 没有要求它做这个区分 |
| 是架构问题 | ✅ 部分是 — 评估官和教练之间的接口设计缺少严重度信号传递 |
| 是输入质量问题 | ❌ 否 |

## 5. 修改动作

### 修改对象

两个文件：

1. `prompts/evaluator_agent.md` — 在输出 Schema 中增加扣分严重度标记
2. `prompts/coach_agent.md` — 增加对"重建型扣分"的处理指令

### 5.1 评估官 Prompt 修改

**修改前** — deduction_points 是纯字符串数组：

```markdown
输出格式:
  deduction_points: ["数据不够精确"],
  hit_points: ["有尝试给出量化数据"]
```

**修改后** — 每条 deduction_point 升级为结构化对象：

```markdown
输出格式:
  deduction_points: [
    {
      "description": "声称做了A/B实验但回答中不存在实验设计——无随机分流方式、
                      无样本量、无对照组、无显著性检验",
      "severity": "foundational",
      "coach_hint": "这不是数据表达问题。候选人需要从零建立A/B实验的方法论基础，
                     而非修改措辞。建议教练给出'面试前学习清单'而非'回答优化技巧'。"
    }
  ],
  hit_points: []

字段说明:
  - severity: "minor"（措辞/格式层面的小问题）
              "moderate"（内容/结构层面的明显缺陷）
              "foundational"（方法论/概念层面的根本性缺失——教练应给出
                             '从零建立'型建议而非'修改措辞'型建议）
  - coach_hint: 给教练 Agent 的定向提示，帮助教练判断该给什么类型的建议。
                仅在 severity="foundational" 时为必填。"
```

### 5.2 教练 Prompt 修改

**修改前** — 教练对所有扣分点采用统一处理策略：

```markdown
每条 coaching_entry 必须对应评估官指出的一个具体扣分点。
```

**修改后** — 增加按严重度分层的建议策略：

```markdown
每条 coaching_entry 必须对应评估官指出的一个具体扣分点。

**建议策略分层（新增）**：

在生成建议前，先读取 deduction_point 的 severity 标记，采用不同策略：

| severity | 教练策略 | 建议语气 | 示例 |
|----------|---------|---------|------|
| "foundational" | **重建型建议**：告诉候选人这个领域需要从零建立基础，给出学习路径和优先级，而非措辞优化技巧 | "你目前在这个领域缺少方法论基础。建议面试前..." | "你描述的'A/B实验'实际只是一次数据查看，不是实验。建议面试前阅读《Trustworthy Online Controlled Experiments》第1-4章，并用一个你熟悉的场景自己设计一遍完整的实验方案。" |
| "moderate" | **改进型建议**：给出具体的改进方法和范例 | "你的方向对了，但执行可以更好。下次可以..." | "你的实验有对照组但未说明分流方式。下次回答时补充：'使用user_id hash分流，实验组和对照组各5000用户，实验运行14天'" |
| "minor" | **微调型建议**：指出措辞或表达细节 | "一个小建议..." | "将'大概15%'改为精确到一位小数的百分比，并注明统计口径" |

**强制规则**：
- 若某条 deduction_point 的 severity="foundational"，教练**禁止**将其对应的 top_3_actions rank 排在"moderate"类建议之后。
- 若 severity="foundational"，`how_to_improve` 字段**禁止**只给出表达层面的建议（如"措辞更精确""表达更有条理"等）。
- 若 severity="foundational"，`example_better_response` 字段必须体现"即使没有实操经验，也可以展示方法论理解"的替代回答策略——而非虚构一个完美的A/B实验案例。
```

### 修改点对比

| 维度 | 原版 | 强化版 |
|------|------|--------|
| 评估官扣分严重度 | 无此概念 | 三级严重度（minor/moderate/foundational） |
| 扣分→教练信号 | deduction_points 纯文本，教练自行理解 | 新增 `coach_hint` 字段定向提示 |
| 教练建议策略 | 统一处理所有扣分点 | 三级分层策略 + 每级有明确模板 |
| 重建型扣分的处理 | 无此概念，教练会当改进型处理 | 强制规则：禁止用表达建议应对方法论缺失 |
| 优先级排序 | 教练自行判断 | 强制规则：foundational 级别的扣分不可排在 moderate 之后 |

## 7. 面试讲述要点

在面试中展示这个 Badcase 时，建议强调：

1. **"我在跨 Agent 接口处发现了信号丢失"** — 这个 Bug 不在任何一个 Agent 内部，而在两个 Agent 的契约层。评估官知道扣分很严重，但没有把严重度显式传递给教练。这是分布式 Agent 系统的典型问题——每个 Agent 独立看都没错，但组合起来就出错。

2. **"我设计了 severity 三级模型"** — 不是简单地加一个字段，而是设计了 minor/moderate/foundational 三级语义，每级对应不同的教练策略。这展示了"模型设计"能力——为 AI 的行为空间定义清晰的分类和路由规则。

3. **"修复是双向的"** — 不只改了教练（让它更好地理解评估），也改了评估官（让它更精确地表达）。这展示了系统性思维——接口问题需要两端一起修。

4. **"这个 Badcase 如果没发现，后果是什么"** — 用户会反复在面试中暴露同一个方法论缺失，但每次都以为"下次把数据说精确点就行了"。产品不仅没帮到用户，还会给用户一种"我已经在改进了"的错觉。
