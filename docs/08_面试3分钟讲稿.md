# 面试 3 分钟讲稿：AI 面试训练 Agent

> **适用场景**：AI 产品经理面试中的"请介绍一个你做过的项目"
> **时长**：约 3 分钟（语速适中，约 550-600 字）
> **侧重点**：用户痛点 → Agent 架构 → 评测闭环 → 迭代能力

---

## 讲稿正文

---

我做了一个 AI 面试训练 Agent，面向的是和我一样的转行求职者——特别是准备面试 AI 产品经理的人。

**先说什么问题。** 我自己在准备面试时最大的痛点是：面试完之后得不到结构化反馈。HR 只会说"不合适"，但不告诉你答得不好在哪里。自己对着镜子练或录音回听，能发现的只是"嗯啊"口头禅，判断不了逻辑对不对、考点有没有命中。而且 AI PM 面试和传统 PM 很不一样——面试官会问 Multi-Agent 协作、Prompt Engineering、AI 产品评估框架这些，你很难找到有经验的人帮你模拟。

**所以我做了一个多 Agent 协作的面试训练系统。** 整条链路是：用户粘贴目标 JD → JD 解析 Agent 提取岗位画像 → 出题官生成定制化面试题和追问树 → 面试官 Agent 和用户多轮对话、根据回答动态追问 → 评估官从四个维度打分（内容匹配度、结构清晰度、证据支撑度、表达可信度，各 0-3 分）→ 教练 Agent 给出逐题改进建议和 Top 3 行动项 → 最后 Judge Agent 做一次独立的质量复核，检查评分是否自洽、建议是否可执行、有没有过度承诺。

**这里面我重点做了两件事。** 第一是 Agent 之间的契约设计。六个 Agent 不是简单的 prompt chain——每个 Agent 有独立的人格设定、输入输出 Schema 和约束条件。评估官扣的每一条分，必须能被教练 Agent 定位到具体的病灶并生成可行动的建议。如果评估官的扣分写成"回答不够深入"这种笼统话，教练就没法给出具体指导。所以我设计了很多约束规则和反例来防止这种问题。

**第二是评测体系。** 我做 AI 产品的一个核心观点是：没有评测就等于拍脑袋调 prompt。我建了一个四维评分 Rubric，每条 0-3 分都有锚定标准和典型表现。然后找同行标注了一批对话，三人交叉标注拉齐后做 Golden 分，再把 AI 评分和人工评分做偏差分析。目前 V1 评测集有 10 条 case，覆盖了三种岗位类型和三个级别。我还做了 Badcase 分类体系——幻觉、评分矛盾、尺度偏移、概念误判，每种有对应的根因分析和修复方案。

**这个项目展示的是什么能力？** 我觉得三点：第一，我能设计 Multi-Agent 系统的架构和契约，不是简单调 API。第二，我知道怎么建评测体系——用数据而不是直觉来驱动 Agent 迭代。第三，我理解 AI 产品的边界——所有报告里都有明确声明"不承诺面试通过率、不替代真实面试官"，我知道什么时候该让 AI 做事、什么时候该坦诚边界。

这个项目还在迭代中，目前 V1 完成了端到端 MVP 和基础评测，后续计划加流式对话和语音交互。如果你感兴趣，我可以展开讲任何一个环节的设计细节。

## 讲稿使用说明

### 三个可以自然停顿的节点

| 节点 | 位置 | 如果面试官此时打断，你可以自然展开 |
|------|------|----------------------------------|
| 节点 1 | "…你很难找到有经验的人帮你模拟" 之后 | → "你为什么不做题库？" → 展开"JD 驱动 vs 题库"的设计决策 |
| 节点 2 | "…防��这种问题" 之后 | → "能具体看看 Agent 之间的契约怎么设计的吗？" → 打开 `prompts/` 目录展示 |
| 节点 3 | "…根因分析和修复方案" 之后 | → "评测集怎么建的，多少条够用？" → 展开评测集规模 vs 质量的讨论 |

### 节奏提示

- **痛点部分**（前 30 秒）：慢一点，让面试官感受到这是真问题而非编的需求
- **链路描述**（中间 60 秒）：不要背书，像讲解白板一样自然。可以边说边用手比划流程
- **评测部分**（后 60 秒）：这是体现 AI PM 核心能力的部分，语气可以稍微自信一些
- **总结部分**（最后 30 秒）：三点总结 + 坦诚迭代中 + 开放追问

### 不需要讲的内容（除非被追问）

- 技术栈细节（FastAPI / React）— 这不是技术面
- 具体的 prompt 文本 — 太细节，会打断讲稿的节奏
- 评测指标的具体数值 — 如果被追问再打开 `evals/eval_report.md` 展示
