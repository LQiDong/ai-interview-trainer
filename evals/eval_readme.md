# 评测体系说明

> **定位**：AI 产品经理面试作品集的评测闭环展示，**不是学术 benchmark**。
>
> 目标不是发论文，而是展示你做 AI 产品时如何**系统化地评估 Agent 质量、发现 Badcase、驱动迭代**。

---

## 1. 评测体系概览

```
                    ┌──────────────────────┐
                    │     Rubric (评分标准)  │
                    │   evals/rubric.md     │
                    └──────────┬───────────┘
                               │ 统一评分尺度
                               ▼
┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐

└──────────────────┘   └──────────────────┘   └──────────────────┘
                                                       │
                                                       ▼
                                              ┌──────────────────┐
                                              │   迭代闭环        │
                                              │ Badcase → 根因    │
                                              │ → 修复 → 回归    │
                                              └──────────────────┘
```

## 2. 评测集来源

### 2.1 数据构成

| 来源 | 说明 | 预计规模 |
|------|------|---------|
| **真实模拟面试** | 邀请 5-8 位转行 AI PM 的求职者使用本产品进行模拟面试，征得同意后收录对话记录 | 20-30 组（对话→评分） |
| **项目作者自测** | 项目作者以不同岗位（AI PM / AI 运营 / Agent PM）和不同水平（初级/中级/高级）身份进行模拟面试 | 10-15 组 |
| **公开面经改写** | 从牛客、知乎、即刻等平台收集 AI PM 面经中的问答片段，改写为对话格式并标注 | 10-15 组 |
| **对抗样本** | 故意构造的边界 Case：完美回答、完全偏题、数据编造、背诵模板等 | 5-10 组 |

### 2.2 数据收录原则

- ✅ **征得同意**：真实面试对话收录前必须获得用户明确同意（提供 opt-in 选项）
- ✅ **脱敏处理**：去除姓名、公司名、项目名等可识别信息，仅保留问答结构和评分标注
- ✅ **覆盖多样性**：确保评测集覆盖 3 个岗位 × 3 个级别 × 4 个维度的交叉组合
- ❌ **不编造**：不凭空捏造"看起来合理"的对话作为评测样本。对抗样本也必须标注为"人工构造"
- ❌ **不抄袭**：不直接复制网上可定位到具体个人的面经全文

### 2.3 字段说明

详见 [`eval_set_template.csv`](./eval_set_template.csv)。

| 字段组 | 包含字段 | 说明 |
|--------|---------|------|
| **题目信息** | `jd_category`, `jd_level`, `jd_text`, `question_id`, `question_text`, `question_type`, `reference_answer_points` | 定义被评估的上下文 |
| **回答信息** | `answer_id`, `answer_text`, `follow_up_chain` | 被评估的实际内容 |
| **标注结果** | `annotator_N_*` (N=1,2,3) | 3 位标注者的独立评分 + 证据引用 + 备注 |
| **Golden 标准** | `golden_*` | 经过拉齐讨论后的最终评分和理由 |
| **元数据** | `difficulty_level`, `has_consensus`, `consensus_notes` | 标注质量和难度标记 |

## 3. 标注规则

### 3.1 标注者资质

| 要求 | 说明 |
|------|------|
| **人数** | 每条样本至少 3 人独立标注 |
| **背景** | 至少 2 人为 AI 产品经理从业者（≥2 年经验），可含项目作者本人 |
| **培训** | 标注前必须完成评分校准流程（见 [rubric.md](./rubric.md) 评分校准规则） |
| **独立标注** | 3 位标注者不得在评分阶段互相讨论或参考对方评分 |

### 3.2 标注流程

```
第 1 步：标注准备
  - 阅读 Rubric 全文
  - 完成校准会议（2 份标杆样本对齐）
  - 确认本次岗位级别参照系
        ↓
第 2 步：独立评分（3 人并行）
  - 阅读对话记录全文
  - 对每道题 × 4 个维度独立打分
  - 写出 evidence_quote（对话原文引用）和 notes（评分理由）
        ↓
第 3 步：一致性检查
  - 检查 IRR（评分者间信度）
  - 标注"分差 ≥ 2"的维度 → 强制讨论
  - 标注"三人三个分数"的维度 → 表决
        ↓
第 4 步：拉齐讨论
  - 对不一致维度逐一讨论
  - 达成一致或由 senior 标注者仲裁
  - 确定 Golden 分 + golden_rationale
        ↓
第 5 步：Badcase 识别
  - AI 评分 vs Golden 分偏差 ≥ 1.0 → 标记为 Badcase
  - Badcase 分类入库（按 Agent + 错误类型打标签）
```

### 3.3 评分规范

| 规范 | 说明 |
|------|------|
| **先说证据再打分** | 每个维度的评分必须先写 `evidence_quote`（对话原文引用），再给分数。这样可以检查评分是否有文本依据 |
| **分差记录** | 标注者之间的评分差异本身就是有价值的数据——高分差区域往往对应 Rubric 的模糊地带，需重点关注 |
| **Golden 分不是绝对真理** | Golden 分是 3 人拉齐后的一致意见，不代表"绝对正确"。如果后续发现 Golden 分有问题，可以修正并记录修正理由。 |
| **标注者署名** | 每条标注记录标注者 ID（可用代号），方便事后追溯和计算单人重测信度 |

## 4. 人工复核规则

### 4.1 复核触发条件

以下情况必须触发人工复核：

| 条件 | 处理方式 |
|------|---------|
| 任意维度两位标注者分差 ≥ 2 | 强制讨论，必须达成一致或仲裁 |
| 三人三个不同分数 | 表决取中位数，若不同意则全组讨论 |
| AI 评估分与 Golden 分偏差 ≥ 1.0 | 标记为 Badcase，分析 AI 评分偏差根因 |
| 标注者 notes 中出现"不确定""存疑"等表述 | 该条样本标记为 `difficulty_level: hard`，优先复核 |
| 同一标注者连续 5 条评分与其他两人偏差 > 0.5 | 提示该标注者可能需要重新校准 |

### 4.2 复核产出

每轮复核产出：

1. **Golden 分更新**（如有修正）
2. **Badcase 列表**（AI 评分偏差 ≥ 1.0 的样本）
3. **Rubric 修正建议**（如果争议集中在某一条 Rubric 描述的模糊地带）
4. **标注者反馈**（对偏离度高的标注者给予具体反馈）

### 4.3 复核频率

| 阶段 | 频率 | 说明 |
|------|------|------|
| MVP 构建期 | 每新增 10 条样本做一轮复核 | 小步快跑，及时发现问题 |
| 稳定维护期 | 每新增 30 条样本做一轮复核 | 规模效应，减少复核开销 |
| Prompt 变更后 | **立即**做回归评测 | 任何 Agent Prompt 的修改必须以回归评测通过为前提 |

## 5. Badcase 到迭代的闭环

### 5.1 分类 → 根因 → 修复

```
Badcase 发现 (AI vs Golden 偏差 ≥ 1.0)
    ↓
[分类] 按 Agent + 错误类型打标签
    │
    ├── 幻觉 (AI 评分引用了对话中不存在的内容)
    ├── 评分矛盾 (分数与 evidence 描述不一致)
    ├── 尺度偏移 (系统性偏高或偏低某一维度)
    ├── 遗漏 (AI 未识别出人类标注者认为明显的扣分点)
    └── 概念误判 (AI 对专业概念的理解与人类不一致)
    ↓
[根因分析] 为什么会出现这个 Badcase？
    │
    ├── Prompt 缺陷（指令不清晰、缺少反例、约束不够具体）
    ├── 上下文不足（对话压缩损失了关键信息）
    ├── 模型能力边界（某些概念理解确实需要更强模型）
    └── 输入质量问题（对话本身就存在歧义）
    ↓
[修复方案] 针对根因设计修复
    │
    ├── 改 Prompt（增加反例、细化锚定标准、补充 Few-shot）
    ├── 调整 Agent 分工（把易出错的子任务独立为 Agent）
    ├── 增强上下文（在评估前注入更多结构化信息）
    └── 升级模型（对特定任务使用更强的模型）
    ↓
[回归评测] 确认修复有效且不引入新 Badcase
    ↓
[更新评测集] 将本次 Badcase 及 2-3 个变体加入评测集
```

### 5.2 Badcase 追踪表

| 字段 | 说明 |
|------|------|
| `badcase_id` | 唯一标识 |
| `eval_id` | 关联评测样本 |
| `agent` | 出问题的 Agent（JD Parser / Evaluator / Coach / Judge） |
| `error_type` | 错误分类（幻觉/评分矛盾/尺度偏移/遗漏/概念误判） |
| `severity` | P0(影响评分结论) / P1(影响评分质量) / P2(轻微偏差) |
| `root_cause` | 根因 |
| `fix` | 修复方案 |
| `status` | open / fixed / verified / wontfix |
| `regression_test_id` | 关联回归评测 ID |

## 6. 不是学术 Benchmark 的声明

### 6.1 本评测体系的定位

> 这是一个 **AI 产品经理的工程化评测方案**，不是学术 benchmark。区别如下：

| 维度 | 学术 Benchmark | 本评测方案 |
|------|---------------|-----------|
| **目标** | 证明模型能力、发表论文 | 发现 Badcase、驱动 Agent 迭代 |
| **评测集规模** | 数千到数万条 | 数十到数百条（MVP 阶段 50+） |
| **标注者** | 众包平台（MTurk/Prolific） | 项目作者 + 同行 AI PM |
| **评测指标** | 追求 SOTA，小数点后两位的竞争 | 追求可行动的发现——"这个 Badcase 告诉我该改什么" |
| **Golden 标准** | 多次标注取众数，追求客观真理 | 3 人拉齐的一致意见，承认主观性 |
| **迭代方式** | 发布 paper → 下一个 benchmark | 发现 Badcase → 分析根因 → 修复 → 回归 → 更新评测集 |
| **对外宣称** | "Our model achieves SOTA on XXX benchmark" | "我们的评测集在 4 个维度上覆盖了 20+ Badcase，已修复 15 个" |

### 6.2 能力边界声明

- ❌ **不声称本评测集具有统计显著性**。MVP 阶段样本量有限（预计 50-80 条），评测结果反映趋势而非精确度量。
- ❌ **不声称 Golden 分是绝对标准**。Golden 分是有限标注者达成的共识，不代表所有面试官都会做出相同判断。
- ❌ **不声称评测结果可以预测真实面试中的表现**。AI 模拟面试与真实面试之间存在系统性差异（压力水平、非语言信息、面试官偏好等）。
- ✅ **本评测体系的价值是工程化的**：它帮助我们发现 Agent 在"什么时候会出错、为什么会出错、怎么修"——这与学术 benchmark 的目标完全不同。

## 7. 评测体系的作品集展示要点

在面试中展示本评测体系时，建议强调以下 3 点：

### 7.1 "我建了一个评测闭环，不是调了调 Prompt"

展示从 Rubric 设计 → 评测集构建 → 标注流程 → Badcase 发现 → 根因分析 → 修复 → 回归评测的**完整链路**。这证明了你不是在"拍脑袋调 Prompt"，而是在用产品经理的方式管理 AI 产品的质量。

### 7.2 "我知道我的 Agent 什么时候会出错"

展示 Badcase 分类体系（幻觉/评分矛盾/尺度偏移/遗漏/概念误判）和具体 Badcase 案例。这证明了你不是在说"我的 AI 产品很好"，而是在说"我知道它的边界在哪里，以及我在如何扩展这个边界"。

### 7.3 "评估本身也需要被评估"

展示 Judge Agent（对评估官的评估）和人工复核（对 Judge 的评估）的双层质量保障机制。这证明了你理解 AI 产品的"自监督"和"人在环路"的平衡。
